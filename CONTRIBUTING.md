## Contributing

Hi there! We're thrilled that you'd like to contribute to this project. Your help is essential for keeping it great.

Contributions to this project are [released](https://help.github.com/articles/github-terms-of-service/#6-contributions-under-repository-license) to the public under the project's open source license.

Please note that this project is released with a Contributor Code of Conduct. By participating in this project you agree to abide by its terms.

## Prerequisites

Before you begin, ensure you have:
- A GitHub account
- Git installed on your local machine
- Familiarity with the project's technology stack (check the repository README for details)

## Submitting a pull request

Follow these steps to contribute code changes:

0. **Fork and clone the repository** - Create your own copy to work with
0. **Configure and install the dependencies** - Run `script/bootstrap` to set up your development environment
0. **Verify tests pass** - Run `script/cibuild` to ensure the project builds and tests pass on your machine
0. **Create a new branch** - Use `git checkout -b my-branch-name` with a descriptive branch name
0. **Make your changes** - Implement your changes, add tests to cover new functionality, and verify all tests still pass
0. **Push to your fork** - Push your branch to your forked repository
0. **Submit a pull request** - Open a PR from your branch to the main repository
0. **Respond to feedback** - Be ready to make changes based on code review feedback
0. **Celebrate** - Pat yourself on the back and wait for your pull request to be reviewed and merged!

## Best Practices

Here are a few things you can do that will increase the likelihood of your pull request being accepted:

- **Follow style guidelines** - Adhere to the project's coding standards and style conventions
- **Write tests** - Include unit tests for new functionality and ensure all tests pass
- **Keep changes focused** - Make one logical change per PR. If you have multiple independent changes, submit them as separate pull requests
- **Write clear commit messages** - Follow [commit message best practices](http://tbaggery.com/2008/04/19/a-note-about-git-commit-messages.html) with a clear subject line and detailed description
- **Update documentation** - If your changes affect user-facing functionality, update the relevant documentation
- **Be responsive** - Respond to code review comments in a timely manner
